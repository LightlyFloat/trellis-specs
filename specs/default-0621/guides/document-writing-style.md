# 文档撰写规范

> 适用范围：所有面向交付的项目文档（部署说明、使用手册、开发说明、README 等）。
> 目的：使文档行文符合企事业单位的正式、严谨标准，可直接交付，无需二次修订文风。
> 核心取向：正式、严谨、流畅，但不过度追求书面化，保持清晰易懂。

---

## 一、六条核心规则

### 1. 减少口语化表达
口语化、随意的措辞改为正式表述。

| 口语化（避免） | 正式（采用） |
|---|---|
| 放在哪个盘、哪个目录都行 | 放在任意目录均可 |
| 干什么 | 用途 |
| 起来后 | 启动完成后 |
| 装不了 | 无法安装 |
| 放行防火墙 | 开放防火墙权限 |

### 2. 禁止括号内补充说明
不使用括号追加解释、补充或注释。需要说明的内容并入正文完整表述。

| 避免 | 采用 |
|---|---|
| 已知限制（如实说明） | 已知限制 |
| 必须留着不关（关窗=停该服务） | 窗口需保留，若关闭窗口则服务停止 |

> 例外：括号用于标注纯客观、不可省略的技术参数时可保留，例如端口号
> 「后端 API（端口 8080）」、命令参数。判断标准：括号内是**技术事实标注**则可留，
> 是**解释性/补充性说明**则并入正文。

### 3. 禁止比喻性、抽象描述
不使用形象化比喻或抽象化的口语描述，改为准确的技术表述。

| 避免 | 采用 |
|---|---|
| 黑窗口 | 命令行窗口 |
| 简单粗暴 | （删除该修饰，直接陈述操作） |

### 4. 禁止调侃性说明
不使用调侃、随意、带个人语气的措辞。

| 避免 | 采用 |
|---|---|
| 别纠结 | （删除，直接给出操作指引） |
| 图省事可以试 | 如需简化操作，可使用 |

### 5. 行文流畅、上下文衔接
避免碎句堆砌，使用完整句子并通过连接词形成因果、递进等逻辑关系。

**改写示例：**

> 避免（碎句）：
> 为什么首选手动按序、而不是一键？ Windows 上批量开窗（一键脚本里的 start）
> 在部分机器上会有窗口起不来、还不报错。按序一个个双击，每个服务自己一个窗口，
> 哪个出问题一眼就看到。这才是最可靠的方式。

> 采用（流畅衔接）：
> 推荐手动按序启动，而非一键启动。原因在于，在 Windows 系统上批量开窗时运用
> 一键脚本中的 start 命令，部分设备会出现窗口无法启动且无报错信息的状况。
> 若选择按顺序逐个双击启动，每个服务独占一个窗口，环节出现问题时也便于排查，
> 因此手动启动是最为可靠的方式。

### 6. 正式但不过度书面
用词正式、准确，但不刻意堆砌生僻或冗长的书面语，以清晰传达为准。

| 标题改写示例 | |
|---|---|
| 还想改代码？ | 代码修改简要说明 |
| 怎么确认起好了 | 如何确认启动完成 |
| 让局域网里别的电脑也能访问 | 使局域网内其他电脑可访问 |
| 手动安装兜底（机器装不了 scoop 时） | 手动安装（机器无法安装 scoop 时） |

## 二、硬约束

1. **只改表达方式，不删技术信息。** 改写过程中，所有端口号、文件路径、命令、
   参数、译名、版本号等技术事实必须原样保留，不得因调整文风而丢失或改动。
2. **面向读者的可操作性优先。** 文档若面向操作人员，步骤的清晰、可执行性高于
   文采。正式化不得以牺牲操作指引的明确性为代价。
3. **术语一致。** 同一概念在全文使用统一称谓，例如统一用「命令行窗口」而非
   交替使用「窗口」「黑框」「终端」。

---

## 三、撰写检查清单

完成文档后，逐条自检：

- [ ] 无口语化、随意措辞（参照规则 1）
- [ ] 无解释性括号补充，已并入正文（参照规则 2）
- [ ] 无比喻、抽象口语描述（参照规则 3）
- [ ] 无调侃、个人语气（参照规则 4）
- [ ] 句子完整、上下文衔接顺畅，无碎句堆砌（参照规则 5）
- [ ] 用词正式准确，未过度书面化（参照规则 6）
- [ ] 技术信息（端口/路径/命令/参数/译名）完整无丢失（硬约束 1）
- [ ] 操作步骤清晰可执行（硬约束 2）
- [ ] 术语全文一致（硬约束 3）
