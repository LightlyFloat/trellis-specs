# Vision MCP 调用规范

> **适用范围**：用户输入包含图片（截图、照片、设计稿等）时的处理流程。

---

## 核心规则

所有图片输入 → **自动调用 vision MCP**，无需用户额外指令。

- `vision_analyze`：具备理解能力，可分析界面语义、推断问题、描述内容。
- `vision_ocr`：仅提取文字，适用于纯文本识别场景。

---

## 工具选择

| 场景 | 工具 | 说明 |
|------|------|------|
| 理解截图内容、判断 UI 问题 | `vision_analyze` | 需要语义理解 |
| 识别截图/照片中的文字 | `vision_ocr` | 只需文本提取 |
| 两者都需要 | 先 `vision_analyze`，必要时补 `vision_ocr` | 理解优先 |

---

## 上下文注入决策

分析用户意图后自行决定 prompt 中是否补充工程上下文：

### 无需上下文

- 纯识图 / 独立描述（"这张图是什么"、"帮我看看这个截图"）
- 图片内容与当前工程无关

### 必须携带相关工程背景

- 代码还原（从截图还原代码）
- 调试（截图中的报错需结合工程定位）
- 对比（截图与当前实现对比差异）
- 定向提取（从截图中提取特定字段/配置用于代码修改）
- 设计修改（UI 截图指导前端修改）

携带方式：在 vision 调用的 `prompt` 参数中注入相关文件路径、组件名、接口字段等工程背景信息，使模型输出可直接用于后续开发动作。

---

## 示例

```
用户：[粘贴一张报错截图] 帮我看看这个错误
→ vision_analyze，prompt 中补充当前模块的技术栈和相关文件路径

用户：[粘贴一张表格截图] 帮我把文字提取出来
→ vision_ocr，无需额外上下文

用户：[粘贴一张 UI 设计稿] 照这个改一下前端
→ vision_analyze，prompt 中补充目标组件路径、当前布局结构
```
